# RadioPad desktop — auto-update signing runbook (DESK-001)

**Status:** Active  **Owner:** Desktop / Security  **Last Updated:** 2026-05-04

The RadioPad desktop shell uses Tauri 2's built-in updater. Updates are signed
with an **ed25519** keypair generated by `tauri signer`. The public key is
compiled into the shipped binary via `tauri.conf.json → plugins.updater.pubkey`;
the private key **never** leaves the build infrastructure.

---

## 1. Generate a signing keypair

Run this **only on the secure build host** (or once locally to bootstrap, then
transfer to AWS KMS / SSM and wipe the local copy).

```bash
tauri signer generate -w ~/.tauri/radiopad.key
```

Outputs:

- `~/.tauri/radiopad.key`           — ed25519 **private** key (encrypted with
  the password you supply).
- `~/.tauri/radiopad.key.pub`       — ed25519 **public** key (safe to commit
  / inject at build time).

> **Never** check the private key into git, copy it to a developer laptop,
> or paste it into a chat / ticket. If you suspect exposure, follow §5
> (key rotation) immediately.

## 2. Build-time environment variables

| Variable | Purpose | Where it lives |
| --- | --- | --- |
| `TAURI_SIGNING_PRIVATE_KEY` | base64-encoded private key body, **or** the file path. Required by `cargo tauri build`. | AWS KMS-encrypted SSM parameter `/radiopad/desktop/signing/private-key`, decrypted into the runner only. |
| `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` | Passphrase that decrypts the private key. | SSM parameter `/radiopad/desktop/signing/private-key-password`. |
| `TAURI_SIGNING_PUBLIC_KEY` | Public key body. Templated into `tauri.conf.json → plugins.updater.pubkey` before build. | SSM parameter `/radiopad/desktop/signing/public-key` (also committed under `desktop/keys/<channel>.pub` for transparency). |

GitHub Actions sources all three through OIDC → an AWS IAM role that has
`kms:Decrypt` + `ssm:GetParameter` and **no** other permissions. See
`.github/workflows/desktop-release.yml`.

## 3. Channel layout

We ship three channels. Each has its own keypair.

| Channel | Endpoint prefix | Audience | Pubkey file |
| --- | --- | --- | --- |
| `stable` | `https://updates.radiopad.local/{{target}}/{{arch}}/{{current_version}}` | All customers | `desktop/keys/stable.pub` |
| `beta` | `https://updates-beta.radiopad.local/{{target}}/{{arch}}/{{current_version}}` | Opt-in beta cohort | `desktop/keys/beta.pub` |
| `canary` | `https://updates-canary.radiopad.local/{{target}}/{{arch}}/{{current_version}}` | Internal QA only | `desktop/keys/canary.pub` |

The shipped binary is pinned to one channel at build time. Switching channels
requires a fresh install (no in-place re-signing of an installed binary).

## 4. Staged-rollout manifest shape

The Tauri updater fetches a JSON manifest from the channel endpoint:

```json
{
  "version": "0.4.2",
  "notes": "DESK-001 / DESK-006 hardening; rulebooks v3.",
  "pub_date": "2026-05-12T16:00:00Z",
  "rollout": {
    "percent": 25,
    "min_previous_version": "0.4.0",
    "geo_allow": ["US", "CA", "EU"]
  },
  "platforms": {
    "windows-x86_64": {
      "signature": "<base64 ed25519 sig over the .msi.zip body>",
      "url": "https://downloads.radiopad.local/desktop/0.4.2/RadioPad_0.4.2_x64-setup.msi.zip"
    },
    "darwin-x86_64":  { "signature": "...", "url": "..." },
    "darwin-aarch64": { "signature": "...", "url": "..." },
    "linux-x86_64":   { "signature": "...", "url": "..." }
  }
}
```

Only `signature` + `url` are consumed by the client; everything else is
informational and consumed by the gateway when computing eligibility per
client.

## 5. Key-rotation procedure

Treat key rotation as a *minor* release event.

1. **Generate new keypair** in the secure build environment (§1).
2. **Publish the new public key** under `desktop/keys/<channel>.pub` and bump
   `tauri.conf.json → plugins.updater.pubkey` to the new value.
3. **Dual-sign** every release for one full minor (e.g. `0.5.x`):
   - Old key signs the manifest as today.
   - New key signs a parallel manifest at `…/<channel>/v2/…`.
   - Clients on the old binary continue to verify with the old key; freshly
     installed clients verify with the new key.
4. **Promote** the new key on the next minor (`0.6.0`): clients now ship with
   only the new pubkey. The old endpoint stays alive for 90 days as a
   read-only safety net.
5. **Retire the old key**: revoke the SSM parameter, schedule the AWS KMS
   key for deletion (30-day window), wipe the encrypted backup, and audit
   the rotation in the security incident log even though it is routine.

Document each rotation in `docs/04-security/secrets-management.md` with the
date, rotating engineer, and the SSM parameter version that became active.

## 6. Security notes

- **Signing key NEVER lives on a developer machine.** Local `cargo tauri build`
  produces unsigned artefacts (a banner appears in the build output). Signed
  builds happen exclusively in CI.
- The CI runner accesses the key through GitHub OIDC → an AWS IAM role with a
  KMS decrypt-only grant. No long-lived AWS credentials are stored anywhere.
- Replay protection: see `docs/04-security/threat-model.md` → *Desktop update
  signing*. The updater rejects any manifest whose `version` is `≤` the
  installed version, and the manifest digest is bound into the signature so
  swapping a different signed payload at the same URL fails verification.
- The public key file in `desktop/keys/` is committed for **transparency only**
  — clients trust the pubkey baked into their own binary, not the one in git.
