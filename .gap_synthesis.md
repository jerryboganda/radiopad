RadioPad is genuinely strong on its **safety/governance/security spine** and **provider/AI-gateway core**, and genuinely thin on **clinical workflow breadth**, **structured data**, and **process-as-code (CI gates, conformance, DR)**.

### The biggest cross-area themes (each blocks many line items)

1. **No structured report data model.** The Report entity is six narrative text sections. This single absence blocks the entire RADS engine (RADS-001/002/006/007/008/012), structured/table/numeric input (RPT-003), bidirectional sync (RADS-007), lesion tracking (COMP-003/004, RADS-005), and most §17.1 clinical entities. **This is the #1 foundational dependency.**

2. **No "explain/provenance" surface.** "Why this suggestion?" (RPT-026, §16.2.7, §16.3 trace), source-span attestation (AI-021/AI-028, §16.5), and the Provenance Inspector (§20.2) all need the same span-trace plumbing that does not exist.

3. **Six clinical workflow modules are entirely absent as features:** Worklist (WL/RPT-029/§24.1), Peer Review (PR-001..010), Teaching Files (TF-001..008), Critical Results (CR-001..010), Evidence/Citation (EV-001..006), and real-time Collaboration (RPT-015/016). Each has no entity, controller, RBAC, audit kind, or page.

4. **CI is a single build+golden-case gate.** No coverage gate, no SAST/DAST/dependency/container scanning, no SBOM/SLSA/Cosign, no Playwright e2e, no claims-lint, no WCAG gate. ~30 §28/§29/§31 items are "the function exists but is not gated/verified."

5. **Process/runtime artifacts are documentation, not code.** DR/HA (§27), capacity (§25), tracing+profiling+status-page (§26.3/.4/.7), PCCP runtime, field-failure intake — thoroughly documented, not built.

6. **Identity is half-built for enterprise.** Strong AuthN; weak AuthZ-at-scale: 6/11 RBAC roles, no step-up, no session limits, no break-glass, no impersonation, no API-key lifecycle, no per-tenant MFA policy.

7. **Standards depth is shallow:** FHIR R4-only (no R5/Bulk), no report→DICOM SR export, no UPS-RS worklist, no LOINC/SNOMED/ICD/RadElement bindings, HL7 outbound is in-memory only.

## Part 3 — Cross-cutting risks

### "Satisfy functionally — do not re-platform" (PRD names OSS/languages that conflict with the locked stack)

The locked stack is **Next.js 16 / ASP.NET Core 8 + EF Core / Tauri 2 / Capacitor 6 / .NET 8 CLI**. The following PRD-named technologies must be satisfied **by function on the locked stack**, not by adopting the named library:

- **Rust `radiopadd` daemon (DMN-001, §24.7)** → the ASP.NET API sidecar is the daemon equivalent. Do NOT build a separate Rust binary; satisfy DMN functions (policy engine, quotas, local inference, audit buffer, health/ready) in-process. Binary-size/RSS budgets (≤35MB / ≤30MB) are **N/A** to .NET.
- **Presidio (PHI scrubbing, AI-020/SEC-010/DESK-010)** → implement a .NET recognizer/scrub service; do not require the Python library.
- **SQLCipher (DESK-005, DMN-007)** → AES-256-GCM at-rest over the local store is the functional equivalent; consider encrypting store metadata to fully close.
- **Keycloak / SPIFFE-SPIRE (§18.1)** → use the existing OIDC/SAML/SCIM stack + .NET mTLS; mesh only if EE multi-service.
- **Yjs CRDT (RPT-015)** → use a .NET/JS CRDT lib over SignalR/WebSocket; the *function* (real-time collab) is what counts.
- **WASM/WASI via Wasmtime (DESK-023, PL-006, DMN-008, MCP-006)** → acceptable to host Wasmtime from .NET/Tauri; OS-process isolation is a partial substitute but capability-limiting must be real.
- **Pyroscope/Loki/NATS/ClickHouse/OpenSearch/Redis/MinIO/Qdrant (§24.2, §26)** → judged on function (tracing, profiling, tiered logs, event bus, columnar analytics, object storage, vector search). Build only when the dependent feature lands; Postgres/local are valid interim substitutes.
- **Lago/Killbill (BILL-013)** → CLAUDE.md effectively makes Stripe the chosen engine; BILL-013's "open engine" intent is **partially out of scope** — close only by adding an `IBillingProvider` abstraction, not a second engine.
- **Vite/React-SPA (§24.6)** → Next.js is the locked substitute; ≤200KB budget, Trusted Types, SRI still apply.
- **gRPC/Go/Rust SDKs (§23.3, §29.2)** → ship TS + Python SDKs from OpenAPI; gRPC/GraphQL only if a real service split or read-heavy need emerges.
- **AES-GCM-SIV (SEC-014), uuidv7 vs uuidv4, blake3 vs SHA-256 (§17.2)** → these are genuine *function* gaps (nonce-misuse resistance, time-ordered ids) — schedule them, but they are crypto-detail, not re-platform.
