
## AI Provider Abstraction + MCP/Tools (PRD v2 §14.9, §14.10)
_RadioPad implements the AI Provider Abstraction (§14.9) and MCP/Tool Integration Layer (§14.10) substantially on the locked stack (ASP.NET Core 8 + EF Core backend, Next.js frontend, .NET CLI). The provider-abstraction core is strong: a per-tenant ProviderConfig registry, IAiProviderAdapter interface with ~13 wired adapters, a single AiGateway choke-point that hard-fails PHI-policy violations before inference, encrypted env:/KMS API-key vaulting, AES-256-GCM OAuth refresh-token vault with rotation, per-provider cost/latency/token/availability telemetry (IAiUsageStore), a composite cost/quality/latency router, sandbox model comparison, retention labels, and a background model-drift watch. The MCP layer is also well-built: a tenant tool registry with admin Submit→Approve→Block lifecycle, least-privilege scopes, default-deny of shell:/fs:/net: scopes, per-call append-only audit with input/output hashes, allowlist connectors, Ed25519 signed-manifest verification with a deny-by-default capability registry, and a sandboxed test runner. Gaps that block 100%: PROV-004 (no explicit equal-or-higher compliance fallback chain — routing just filters eligibility), PROV-012 (no provider conformance test suite), PROV-013 (only a MONTHLY plan quota exists; the per-tenant per-DAY USD cost ceiling with hard cutoff + 80% warning is absent), PROV-014/015 (latency-budget and cheapest-compliant are weighted factors, not the specified hard-budget/mode switches), PROV-016 (no structured BAA/DPA/ZDR/training-on-data compliance matrix — only a single compliance class + free-text retention label), PROV-018 (per-modality default override not implemented; modality param is accepted but unused), PROV-019 (BYO-OAuth-app not surfaced), PROV-020 (no confidential-computing attestation). On MCP: MCP-006 sandbox is in-process only, NOT WASM/WASI (explicitly deferred in code), MCP-008 (no tool conformance suite), MCP-009 (no tool usage quotas), MCP-011 (built-in safe tools — PubMed/ACR/ICD-CPT/Fleischner/RECIST — are manifest-only declarations with no implemented connector binding classes), MCP-012 (no explicit prompt-injection-resistant prompts/output validation layer for tools)._

- [PARTIAL] (P0/CE/M) **PROV-004 Fallback routing only between providers of equal-or-higher compliance class** — gap: There is no explicit multi-provider fallback CHAIN that, on a primary failure, re-routes only to an equal-or-higher compliance-class provider. AiGateway invokes a single selected adapter and throws on failure (no automatic failover). Eligibility filtering exists, but the directed 'fallback only to >= compliance' behavior on transport failure is not implemented.
- [PARTIAL] (P0/CE/S) **PROV-009 Data retention labeling per provider and endpoint** — gap: Label is a single free-text string per provider, not a structured per-endpoint retention record, and it is informational only (never enforced). PRD says 'per provider AND endpoint'; per-endpoint granularity and any policy effect are absent.
- [PARTIAL] (P1/CE/M) **PROV-011 First-class provider adapters (OpenAI, Azure, Anthropic, Vertex, Bedrock, Cohere, Mistral, Groq, Together, Fireworks, Replicate + local OSS llama.cpp/Ollama/vLLM/LocalAI/TGI/Tabby)** — gap: Dedicated first-class adapters for Cohere, Mistral La Plateforme, Groq, Together, Fireworks, Replicate are NOT individually implemented (only reachable generically via openai-compatible). Local OSS LocalAI, TGI, and Tabby have no dedicated adapter (also only via openai-compatible). Generic reachability partially satisfies the function but not the 'first-class adapter' clause per named provider.
- [MISSING] (P1/CE/L) **PROV-012 Provider conformance test suite** — gap: Build a runnable, provider-agnostic conformance test suite (golden prompt/response contract checks) any new adapter can be run against to certify RadioPad compatibility.
- [MISSING] (P0/CE/L) **PROV-013 Cost ceiling per tenant per day with hard cutoff; warning at 80%** — gap: Implement a per-tenant per-DAY USD spend ceiling that hard-cuts at 100% and warns at 80%. Current quota is monthly call/token-based, not a daily dollar ceiling, and has no warning threshold.
- [PARTIAL] (P1/CE/M) **PROV-014 Latency budget routing (pick fastest compliant provider under N ms)** — gap: There is no hard latency-BUDGET filter (a configurable N-ms ceiling that excludes slower providers and then picks the fastest compliant one). Latency only influences a soft weighted score, not a budget cutoff mode.
- [PARTIAL] (P2/CE/M) **PROV-015 'Cheapest-compliant' routing mode for non-clinical drafts** — gap: No explicit selectable 'cheapest-compliant' MODE (a switch that, for non-clinical drafts, ignores quality/latency and deterministically picks the lowest-cost compliant provider). - [PARTIAL] (P0/CE/M) **PROV-016 Provider compliance matrix view for admins (BAA, DPA, ZDR endpoints, training-on-data)** — gap: No structured compliance-matrix data model or view with discrete BAA-status, DPA-status, ZDR-endpoint, and training-on-data columns. - [MISSING] (P1/CE/L) **MCP-011 Built-in safe tools: PubMed, ACR Appropriateness, ICD/CPT, Fleischner/Lung-RADS, RECIST helper** — gap: Implement the named built-in safe MCP tools as runnable connector binding classes (PubMed search, ACR Appropriateness lookup, ICD/CPT lookup, Fleischner/Lung-RADS calculator, RECIST helper). - [MISSING] (P0/CE/L) **MCP-012 Prompt-injection-resistant tool prompts and output validation** — gap: Add prompt-injection-resistant tool prompt templates and structured output validation/sanitization for tool results before they re-enter the model/report context. Currently absent for the MCP tool path.

## AI Safety & Quality / Validation Engine (PRD v2 §16)
_RadioPad implements a solid deterministic core of §16.1's validation matrix plus the human-in-the-loop primitives that protect signing, but a large fraction of the PRD's checks and safety-process controls are missing or only partially built. The deterministic engine (backend/RadioPad.Api/src/RadioPad.Validation/Engine/ReportValidator.cs) genuinely covers laterality, measurement, required-section, negation (NegEx-style), modality-mismatch, critical-finding language, and follow-up allow-list checks, all driven by YAML rulebooks (RulebookSpec) and exercised by golden-case validation packs (ValidationPackService) — these map well to the locked .NET stack regardless of the PRD's NER/Go/Rust wording. The "unsupported-impression / hallucination" check exists but is a deterministic token-overlap heuristic (HallucinationDetector.cs), NOT an AI-assisted, source-span-attested check as §16.1/§16.5 require. Roughly a third of the §16.1 matrix has NO implementation at all: anatomy-mismatch, sex/age-appropriate language, allergy/contrast safety, dose (DLP/CTDIvol) language, spelling/grammar (Hunspell/LanguageTool), acronym expansion, prior-report inconsistency (semantic diff), quantitative RECIST comparison, and standalone ambiguity detection. §16.2 human-in-the-loop is partial: AI text is visibly marked (.ai-mark) and acknowledged, export is gated behind explicit acknowledgement and blockers, and the system never auto-signs (RIS copy via CopyToRisButton) — but there is NO critical-results routing workflow, NO keystroke-aggregation analytics, NO no-silent-retry operator-visibility guarantee, NO accessibility opt-out for the AI mark, and NO always-accessible mandatory "Why this suggestion?" panel. §16.3 confidence presentation is partial (pass/warn/block + unsupported-statement warnings + quality score) but lacks source-span highlights, a provenance/context-trace graph, a visible model+rulebook version pin in the panel, and a "Needs radiologist review" banner. §16.4 safety-RFC has tooling scaffolding (validation packs with approve/run/audit) but no enforced RFC/CSWG/≥99%-golden-gate/30-day-pre-announce process. §16.5 adversarial robustness is largely absent: no prompt-injection stripping in the dictation/daemon path, no output watermarking, no source-span attestation, only a weak refusal fallback, and no toxic-content-filter toggle. Evidence verified against actual source; PROGRESS.md claims around §16 (e.g. Prompt Studio for §16.4) were cross-checked and do not substitute for the missing checks._

## Auth, Tenancy & Access Control (PRD v2 §14.1, AUTH-001 … AUTH-015)
_RadioPad has a strong, well-tested authentication core on the locked .NET 8 / Next.js 16 stack, but the §14.1 access-control surface is roughly half-complete. SOLID/implemented: SAML 2.0 SP with signed-assertion validation (SamlController), OIDC Authorization Code + PKCE (AuthController + OidcBearerMiddleware), passwordless magic-link (MagicLinkController) with chained per-email/per-IP rate limiting, TOTP MFA (MfaController, RFC 6238) and WebAuthn/FIDO2 passkeys (WebAuthnController, hand-rolled attestation parser), RFC 8628 device flow (DeviceAuthController), full SCIM 2.0 Users+Groups+role projection (ScimController), account lockout + global session revocation via SessionEpoch (LockoutPolicy + UsersController), and per-tenant + global IP allowlist (IpAllowlistMiddleware). Stateless HMAC bearer tokens with a 12h lifetime (RadioPadBearerTokens), surfaced through AuthSession inventory rows. The frontend has /login, /admin/sso, and /admin/security pages backing these. GAPS: email/PASSWORD sign-in is not wired (PasswordHash field exists but no verify endpoint — only dev-header signin + passwordless flows); RBAC defines only 6 of the 11 PRD roles (missing Resident, Fellow, Subspecialist, Researcher, Auditor/read-only); tenant isolation is row-level TenantId only (no per-tenant DB schema / storage prefix / cache namespace / queue subject); MFA enforcement is a global OIDC env flag rather than a per-tenant policy; and several enterprise controls are entirely absent in code (step-up auth AUTH-008, per-role/per-tenant session limits & idle timeouts AUTH-009, break-glass AUTH-010, service-to-service short-lived JWT + mTLS AUTH-011, tenant-facing API-key lifecycle AUTH-012, geo-restrictions AUTH-014, support impersonation AUTH-015), device trust AUTH-007 captures only an optional fingerprint with no signed cert / posture check. Several gaps are documented in docs/ and ADR-0004 but not built. Effort is judged on the LOCKED stack (function, not the PRD's named OSS library)._

Password as an interactive credential is unimplemented despite the field existing. ## Data &amp; Audit Model (PRD v2 §17)
_RadioPad implements the spine of §17 — a per-tenant, append-only, hash-chained AuditEvent log with a working VerifyChain endpoint (UI + API + CLI), a retention worker honoring RetentionDays + LegalHold, and an AiRequest usage ledger that already stores input/output as hashes only. However, the implementation is materially shallower than the PRD specifies in three areas. (1) §17.1 entity catalog: ~13 of 21 named entities exist; six clinical/longitudinal entities (Critical Result, Peer Review, Teaching Case, Citation, Worklist Item, Lesion) and dedicated AI Response / Validation Result / Report Final entities are absent. (2) §17.2 audit schema: the rich structured event (uuidv7 event_id, ISO event_time, actor{auth_method}, subject{hashes}, ai{model_version,token counts,phi_class,input/output blake3}, validation[], chain{prev/this}, ed25519 signature) is collapsed into a flat {Action enum, DetailsJson} row; hashing is SHA-256 not blake3, there is NO per-event ed25519 signature, NO RFC 3161 timestamp anchoring, and NO Rekor-style transparency log. (3) §17.3 retention: retention is a single tenant-wide RetentionDays value, NOT configurable per data class; HashOnlyAuditMode is a stored/settable flag that is never enforced at write time; there is no right-to-delete workflow; the NDJSON/CEF export exists but is NOT a signed bundle. Per the CLAUDE.md stack-lock rule, items are judged on function (ed25519/blake3/Rekor naming is not penalized for being absent libraries — they are penalized because the FUNCTION, tamper-evident per-event signing and external anchoring, genuinely does not exist on any stack)._

- [PARTIAL] (P2/Core/L) **§17.1 Device entity (studio install / daemon, signed cert)** — gap: No first-class persistent Device entity with a signed device certificate; device identity is a fingerprint hash on auth/session rows, not a cert-bound enrolled device record. No X.509/signed-cert device attestation.
- [PARTIAL] (P2/Core/XL) **§17.1 Report Draft (CRDT document)** — gap: Draft is a plain mutable row, NOT a CRDT document — no conflict-free replicated data type / collaborative merge structure. Multi-editor concurrency relies on versions/signatures, not CRDT.
- [PARTIAL] (P1/Core/S) **§17.1 Report Version (snapshot per edit/export with content hash)** — gap: ReportVersion has NO content-hash field — PRD requires a per-version content hash. Snapshot is stored but not cryptographically fingerprinted on the row.
- [PARTIAL] (P1/Core/M) **§17.1 Report Final (immutable export record)** — gap: No dedicated immutable 'Report Final' export-record entity capturing the exact exported artifact + destination + content hash. Finalization is inferred from status + signatures; the immutable export snapshot record does not exist as its own entity.
- [PARTIAL] (P3/Core/S) **§17.1 AI Response (output, token counts, latency, cost)** — gap: No separate AI Response entity; response is folded into AiRequest. Per-row cost is not persisted on the AI row (computed at summarise time). Acceptable functionally but diverges from PRD's distinct entity + stored cost.
- [PARTIAL] (P2/Core/M) **§17.1 Validation Result (per check, severity, outcome, span refs)** — gap: Validation results are not persisted as a dedicated entity with span references; they are recomputed and embedded in audit DetailsJson / responses. No stored per-check row with span anchors for longitudinal querying.

Immutability is by-convention (no DB trigger / append-only constraint preventing UPDATE/DELETE).
- [MISSING] (P1/Core/L) **§17.1 Critical Result (notification, ack, escalation chain)** — gap: Entire critical-results notification/acknowledgement/escalation-chain entity and workflow is absent.

## Desktop App / RadioPad Studio (PRD v2 §14.7, DESK-001 … DESK-025)
_RadioPad Studio is a Tauri 2 shell (desktop/src-tauri) that loads the static Next.js export and supervises the bundled .NET API sidecar. The security/data spine of §14.7 is genuinely built and reasonably solid: AES-256-GCM-encrypted offline drafts + local cache keyed off the OS keyring, a sha256+ed25519 plugin/model verifier with OS-level sandbox launch, a PHI-redacting tracing layer, a global-hotkey set, secure-clipboard with TTL + clear-on-blur, RFC-8628 device pairing (backend DeviceAuthController + /pair UI + keyring token), a signed PACS-plugin manifest loader, and ed25519 auto-update + per-OS code-signing documented in runbooks. About 11 of the 25 line items are implemented or close; ~4 are partial; ~10 are entirely missing. The biggest gaps are the day-to-day UX surfaces a reporting-room hero is judged on: there is NO tray/menu-bar daemon control + provider switcher (DESK-014, P0), NO OS-native notifications (DESK-013), NO multi-monitor floating overlay or 380x600 mini-panel (DESK-011/DESK-024), NO crash reporter (DESK-017), NO accessibility/AT compatibility work (DESK-018, P0), and NO power-saver, Wacom/pen, foot-pedal, locale hot-reload, per-user themes/high-contrast, or fleet management.
